# GitHub 基本用語まとめ

## 基本用語

### リポジトリ（Repository）
プロジェクトの保管庫。コード・ファイルをまとめて保存する場所。
→ 新しいアプリを作り始めるときに作る

### コミット（Commit）
変更の保存記録。「ここまでの変更を記録しておく」というセーブポイント。
いつ・誰が・何を変えたかが残る。
→ コードを書いたり修正したときにこまめに記録する

### ブランチ（Branch）
作業用の分岐コピー。本番のコード（main）を壊さずに、別の場所で自由に作業できる。
→ 新機能を追加するとき、バグを直すときに作る。後からいつでも作れる。

### マージ（Merge）
ブランチの統合。変更を本番（main）に合体させる。
→ ブランチの作業が終わって本番に反映したいときに実行

### デプロイ（Deploy）
サーバーにコードを載せて動かすこと。ユーザーが使えるようにする。
→ Vercelに繋いでいれば、プッシュで自動デプロイされる

### リリース（Release）
「v1.0完成！」とバージョンに名札をつけること。
→ 小規模なら正直なくても困らない

### Issue（イシュー）
タスク・バグの管理チケット。
→ やるべきことを整理・共有したいときに作る

### Pull Request（PR）
変更のレビュー依頼。チームで確認してからマージする。
→ 一人開発ではあまり使わない

## 一人開発で覚えること

1. **コミット** → こまめに保存
2. **ブランチ** → 実験したいとき（必要に応じて）
3. **マージ** → ブランチを本番に合体
4. **デプロイ** → 本番に反映（Vercelが自動）

## Claude Code での使い方

- 「コミットしてプッシュして」→ ローカル保存＋GitHub送信
- 「GitHubに保存して」「GitHubに上げて」でもOK
- `/commit` スキルで一発操作も可能

## コミット vs プッシュ

- コミット → PCの中に保存（ローカル）
- プッシュ → GitHubに送信（クラウド）

## GitHub MCP

- 自動では何もしない。指示したときだけ動く
- リポジトリ作成、Issue、PR作成などGitHub上の操作ができる
