## Claude
GitHub MCP でできる主な操作をざっくりまとめます：

### リポジトリ操作
- リポジトリ作成 → `create_repository`
- リポジトリ検索 → `search_repositories`
- ファイル取得/作成/削除 → `get_file_contents`, `create_or_update_file`

### Issue（課題管理）
- Issue一覧/検索 → `list_issues`, `search_issues`
- Issue作成/更新 → `issue_write`
- コメント追加 → `add_issue_comment`

### Pull Request
- PR作成 → `create_pull_request`
- PR一覧/検索 → `list_pull_requests`, `search_pull_requests`
- PRレビュー → `pull_request_review_write`
- PRマージ → `merge_pull_request`

### ブランチ・コミット
- ブランチ作成/一覧 → `create_branch`, `list_branches`
- コミット一覧/詳細 → `list_commits`, `get_commit`

### その他
- リリース管理 → `list_releases`, `get_latest_release`
- コード検索 → `search_code`
- ユーザー検索 → `search_users`

使い方の例：自然に日本語で頼むだけでOK

- 「新しいリポジトリ `my-app` を作って」

- 「このリポジトリにIssueを3つ作って」
- 「PRを作ってマージして」
- 「GitHubで○○のコードを検索して」

## ユーザー
それぞれの言葉の意味を教えて。どういう時に使うの？

## Claude

### リポジトリ（Repository）
= プロジェクトの保管庫。コード・ファイルをまとめて保存する場所。フォルダみたいなもの。
→ 「新しいアプリを作り始めるとき」に作る

### コミット（Commit）
= 変更の保存記録。「ここまでの変更を記録しておく」というセーブポイント。いつ・誰が・何を変えたかが残る。
→ 「コードを書いたり修正したとき」にこまめに記録する

### ブランチ（Branch）
= 作業用の分岐コピー。本番のコード（main）を壊さずに、別の場所で自由に作業できる。
→ 「新機能を追加するとき」「バグを直すとき」に作る

### Pull Request（PR）
= 変更のレビュー依頼。「こういう変更をしたので、本番に取り込んでください」という申請。
→ 「ブランチの作業が終わって、本番に反映したいとき」に作る

### マージ（Merge）
= ブランチの統合。PRが承認されたら、変更を本番（main）に合体させる。
→ 「PRがOKになったとき」に実行する

### Issue（イシュー）
= タスク・バグの管理チケット。「この機能を作りたい」「このバグを直したい」をチケットとして記録。
→ 「やるべきことを整理・共有したいとき」に作る

### リリース（Release）
= 公開バージョン。「v1.0」「v2.0」のように、完成した状態に名前を付けて公開する。
→ 「アプリの新バージョンを公開するとき」に作る

### 一人で使う場合の流れ
1. リポジトリを作る（プロジェクト開始）
2. コードを書いてコミット（こまめに保存）
3. そのまま main にプッシュ（一人なのでPR不要なことも多い）

### チームで使う場合の流れ
1. Issue を作る（「ログイン機能を追加」など）
2. ブランチを作る（main を壊さず作業）
3. コードを書いてコミット
4. PR を作る（チームにレビュー依頼）
5. レビューOK → マージ（本番に反映）
6. Issue を閉じる（完了）

## ユーザー
本番バージョンはコミットだけでいいか。覚えるのはコミットとブランチとマージ。基本的にコミットをたくさん使っていくって感じか。基本的にそのコミットした内容をリリースしていくってこと。リリースとデプロイって何が違うの？

## Claude

### デプロイ（Deploy）
= サーバーにコードを載せて動かすこと。「このコードを本番サーバーに反映して、ユーザーが使えるようにする」
→ Vercelにプッシュしたら自動でデプロイされる、あれのこと

### リリース（Release）
= 「v1.0 完成！」とバージョンに名札をつけること。GitHubの機能。

ぶっちゃけた違い：
- デプロイ → 実際に動かす（実作業）
- リリース → 名前をつけて記録する（ラベル貼り）

一人開発ならデプロイだけ覚えればいい。

まとめると覚えるのは：
1. コミット → こまめに保存
2. ブランチ → 実験したいとき（後から必要に応じて）
3. マージ → ブランチを本番に合体
4. デプロイ → 本番に反映（Vercelが自動でやってくれる）

## ユーザー
このGitHubのMCPはいつ作動するの?デフォルトで保存してくれてるとこはないよね

## Claude
自動では何もしません。全部こちらから指示したときだけ動きます。

- コードを書く → Claude が書く（ローカルのファイル）
- コミットする → 「コミットして」と言う → git で保存
- GitHubにプッシュ → 「プッシュして」と言う → GitHub に送信
- リポジトリ作成 → 「リポジトリ作って」と言う → GitHub MCP が作る
- Issue/PR作成 → 「Issue作って」と言う → GitHub MCP が作る

ローカルのgit操作（コミット、プッシュ）→ Bashのgitコマンド
GitHub上の操作（リポジトリ作成、Issue、PR）→ GitHub MCP

どっちも言われたときだけ動く。勝手にバックアップしてくれる機能はない。

## ユーザー
ブランチって後から作れるの？1人でやる場合って、リポジトリで管理をして、コミットだけでいいんじゃないかな。コミットだけでいけるんじゃない？

## Claude
ぶっちゃけ、その通り。一人ならコミットだけでほぼ OK。

ブランチは「同時に2つの方向で作業したいとき」に便利。

一人開発の現実的な使い分け：
- 普通にコード書いて保存 → ブランチ不要。コミットだけ
- 過去に戻ってやり直す → ブランチ不要。コミットで戻ればいい
- 大きな実験をしたい（壊すかも）→ あると安心。失敗したら捨てるだけ
- Vercelにデプロイしてる → あると便利。main=本番、別ブランチ=開発中

覚えるのはコミットだけでいい。ブランチは「あ、これ分けたいな」と思ったときに後からいつでも作れる。
